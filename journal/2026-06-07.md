## 2026-06-07

### 新闻候选池和 DeepSeek 稳定性
- 现象：用户反馈过去 24 小时候选新闻明显过少，且 DeepSeek 经常因缺标题、10 屏结构不严格或正文字数不在 1400-1600 范围内而降级。
- 判断：候选池对每个来源先做严格 24 小时过滤，Hugging Face、arXiv、GitHub 这类非实时流容易被清空；Hacker News 又只保留企业关键词命中，导致可送入 DeepSeek 的素材偏少。
- 处理：每个来源仍优先使用 24 小时内条目，但当过滤后少于最小保留量时，会用该来源已抓到的最新候选回填；Hacker News 改为企业相关优先排序，但不再丢弃普通 AI 讨论。
- 判断：DeepSeek 返回内容经常是可修复的轻微偏差，原逻辑直接降级，导致兜底文章比例过高。
- 处理：DeepSeek JSON 归一化增加修复层：缺顶层标题时从具体板块标题推导；正文略短时用兜底文章片段补足到发布字数；正文过长时截到 1600 字以内；不再截断模型返回的多段正文。
- 检查：`npm.cmd run test:run`、`npm.cmd run lint`、`npm.cmd run build --cache C:\tmp\npm-cache` 均通过。

### 公众号定位放宽与 DeepSeek 输入压缩
- 现象：用户采纳“不纯做企业落地”的建议，并询问 DeepSeek 输出被截断的原因。
- 判断：截断原因是 prompt 过长且输出任务过重；系统最多送入 100 条素材，每条摘要 360 字，同时要求 DeepSeek 生成 10 屏 JSON、配图提示词和 1400-1600 字正文，容易碰到 `max_tokens=8000` 输出上限。
- 处理：公众号定位从“企业 AI 落地”放宽为“AI 应用与产业信号”，覆盖产品进展、模型能力、开源生态、开发者工具、多模态应用、产业动态和商业化变化。
- 处理：送入 DeepSeek 的素材压缩为最多 45 条，每条摘要最多 180 字，降低输出截断概率。
- 处理：候选排序补充 model、multimodal、video、open source、release、product、benchmark 等 AI 应用关键词，避免只偏向企业流程词。
- 检查：`npm.cmd run test:run`、`npm.cmd run lint`、`npm.cmd run build --cache C:\tmp\npm-cache` 均通过。

### 生成仍少与 DeepSeek 截断二次修复
- 现象：用户再次生成时送入 DeepSeek 的候选仍只有 18 条，且第二次 DeepSeek 返回被 `max_tokens=8000` 截断。
- 判断：AI HOT 的 fallback 函数实际没有 fallback，只抓 24 小时 selected 数据；如果当天少，源头不会补抓更宽窗口。DeepSeek 侧仅压缩输入还不够，输出上限仍偏紧。
- 处理：AI HOT 在 24 小时素材少于最小数量时，自动回补 7 天 `mode=all` 数据；聚合层如果任一来源使用 7 天回补，整体 sourceWindow 标为 `7d`，避免文章误写“过去 24 小时”。
- 处理：DeepSeek `max_tokens` 从 8000 提高到 12000，并在 prompt 中要求输出紧凑 JSON、每屏 body 1-2 段，减少冗余输出导致的截断。
- 检查：`npm.cmd run test:run`、`npm.cmd run lint`、`npm.cmd run build --cache C:\tmp\npm-cache` 均通过。
